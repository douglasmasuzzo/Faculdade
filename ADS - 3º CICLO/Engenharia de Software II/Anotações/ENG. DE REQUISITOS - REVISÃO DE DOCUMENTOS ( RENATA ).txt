  - CORRIGIR A TABELA ( 3.1 - RESUMO DOS ENVOLVIDOS ), INFORMANDO A ÁREA DE DESCRIÇÃO ( CARGO | FUNÇÃO ), ÁREA DE RESPONSABILIDADE ( DESCRIÇÃO )...SEGUINDO A PADRONIZAÇÃO ESTABELECIDA PELA IBM.

  - NO AMBIENTE DO USUÁRIO ( 3.2 ), INFORMAR MAIS SOBRE TÓPICOS TÉCNICOS ( FERRAMENTAS \ RECURSOS ) ENTRE OUTROS. ( EXEMPLO VISUAL: VISÃO_SISTEMA_PROPOSTA_TPC )

  - NO DOCUMENTO DE VISÃO, NÃO É NECESSÁRIO DEFINIR TODOS OS REQUISITOS 

  - PALAVRAS CHAVES QUE PODEM SER ADICIONADAS NA DESCRIÇÃO DO RF: GERENCIAMENTO | MÓDULOS | GESTÃO

  - CORRIGIR O RF14 ( EMISSÃO DE COMPROVANTE DE COMPRAS ) E ANEXAR AO RF10 ( CONTROLE DE VENDAS ). A DESCRIÇÃO DO RF14 ( USO DE API ) DEVE SER REDIRECIONADO A DESCRIÇÃO DO RF13 ( MÉTODO DE PAGAMENTO ) 

  - A DESCRIÇÃO DE RF14 ( EMISSÃO DE COMPROVANTE ) PODE TAMBÉM SER DIRECIONADO AO USUÁRIO, COM A RF07 ( COMPRA ONLINE ).

  - OS 20 RF'S DESSE PROJETO PODE ACARRETAR EM 300 / 400 AÇÕES OPERACIONAIS. EM UMA CORPORAÇÃO, ESTIMA-SE A EXISTÊNCIA DE +100 MATRIZES.

  - REFERENTE AO REQUISITO NÃO-FUNCIONAL ( RNF032 -  INTEGRAÇÃO COM REDES SOCIAIS ), DEVE SER REDIRECIONADO COMO UMA AÇÃO FUNCIONAL ( SUBSTITUIR O RF14 ). 

  - SOBRE OS RNF, EXPLORAR MAIS O AMBIENTE DO USUÁRIO ( ÁREA TÉCNICA ) LADO DO USUÁRIO INFORMAR QUAIS SÃO AS FERRAMENTAS, OS RECURSOS, AS FUNCIONALIDADES, ENTRE OUTROS

  - SOBRE O RNF, PODEMOS INCLUIR QUESTÃO DE IDIOMA PADRONIZADO DO SISTEMA | EFICIÊNCIA DE DADOS

** CASOS DE USO - ( HENRY )

   - A EXIBIÇÃO DO PRODUTO RELACIONA-SE A EXISTÊNCIA NÃO OBRIGATÓRIA AO ESTOQUE, PODENDO SER INFORMADO PELA " EXISTÊNCIA " DENTRO DO ESTOQUE, E CONSUMIDO E SOLICITADO PELO GERENCIAMENTO DE ESTOQUE ( INTERLIGADO AO DEPARTAMENTO DE VENDA )

   - CONSIDERAR A IMPORTÂNCIA DO RELATÓRIO DE COMPRA PARA SOLICITAÇÃO DO PRODUTO SOB DEMANDA. 

   - O " CADASTRAR PRODUTO " DEVE CONSIDERAR A ADIÇÃO, EDIÇÃO E DESATIVAÇÃO DO PRODUTO DENTRO DO ESTOQUE ( CRUD DO PRODUTO ) 

   - DENTRO DO DIAGRAMA, DEVE DIFERENCIAR A AÇÃO DO MÉTODO SOBRE A AÇÃO DO PROCESSO

   - TÉCNICA : PODE AGRUPAR RECURSOS EM UM ÚNICO BLOCO DE DESCRIÇÃO DO CASO DE USO 

   - CADA TÓPICO REFERENTE AO CASO, DEVE SER REPRESENTADO NO DIAGRAMA

    -- GESTÃO DE ESTOQUE

    + ATOR : ADM
    + INCLUDE : REGISTRAR ENTRADA | REGISTRAR SAÍDA 
    + FUNÇÃO QUE PODE SER HERDADA DE OUTRO ATOR 


** ESPECIFICAÇÃO DE CASO DE USO - FUNCIONALIDADE

   - DEFINIÇÃO DE TELA : QUANTIDADE, FUNCIONALIDADES, RECURSOS E ETC 
  
   - BUSCAR DISTRIBUIR AS FUNCIONALIDADES DA TELA, ONDE CADA TELA POSSUI SUA IMPORTÂNCIA

   - AO CITAR O TIPO DO ATOR, DIGITE APENAS O " ATOR "

   - NO FINAL DE TODO FLUXO, APÓS DESCREVER DETALHADAMENTE O CASO DE USO, MENCIONE " CASO DE USO SE ENCERRA "
    ++ " Caso de Uso se encerra "

   - AO INCLUIR OUTRO CASO DE USO EXTERNO, DIFERENTE DA ESPECIFICAÇÃO, DEVE SER MENCIONADO A EXECUÇÃO DO TIPO DE UC INFORMADO
    ++ " ex: O sistema executa o UC - Enviar Notificação "
    ++ DIAGRAMA: ( UC < include > UC ) 


   - SE HAVER A NECESSIDADE DE CRIAR OUTRO CASO DE USO QUE INCORPORE OUTROS RECURSOS DEFINIDOS EM CASOS DE USO, A CRIAÇÃO DEVE SER IMPLEMENTADA NO DIAGRAMA E " < INCLUIR > " A EXECUÇÃO DO CASO DE USO ESPECIALIZADO  
   
   - RELEMBRANDO: REGRA DE NEGÓCIO NÃO É CASO DE USO

   - DIFERENCIAR A REGRA DE UM PROCESSO EM RELAÇÃO AO CASO DE USO 

   - A DEFINIÇÃO DE UMA REGRA SE INTRELAÇA A EXISTÊNCIA DE MÉTODO
